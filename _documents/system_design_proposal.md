# システム構成案: 動画サブスクリプションサイト

## 概要
既存の「システム構成図」にあるツールシステムに加え、サブスクリプション会員（有料会員）のみが視聴可能な動画解説サイトを追加するためのシステム構成案です。

## 前提条件・現状のスタック
- **Frontend/Auth**: Next.js 15 (App Router), Supabase Auth, Tailwind CSS (`dataviz-auth`)
- **Backend/API**: Node.js, Stripe, Supabase DB (`dataviz-api`)
- **CMS/LP**: Hugo (`dataviz-jp`)

## 要件
1.  **アクセス制御**: 有効なサブスクリプションを持つユーザーのみがアクセス可能。
2.  **動画配信**: ストリーミング配信（ダウンロード不可が望ましい）、快適な再生。
3.  **UI/UX**: モダンで使いやすい動画一覧・再生画面。

---

## 提案1: アプリケーション・アーキテクチャ

### Option A: 既存 `dataviz-auth` (Next.js) への統合 【推奨】
現在 `dataviz-auth` が Next.js で構築されており、認証(Supabase)とUIライブラリ(Shadcn UI)がセットアップ済みです。これを拡張して動画ページを追加するのが最も低コストで保守しやすい方法です。

*   **メリット**:
    *   認証状態を共有しやすい（SSOなどの実装不要）。
    *   既存のUIコンポーネントを再利用できる。
    *   デプロイ・管理するプロジェクトが増えない。
*   **デメリット**:
    *   認証アプリが肥大化する可能性がある（ディレクトリ構成で分離すれば緩和可能）。

### Option B: 新規 Next.js アプリケーションの作成
動画サイト専用に別の Next.js プロジェクトを立ち上げる構成です。

*   **メリット**:
    *   コードベースが分離され、責務が明確になる。
*   **デメリット**:
    *   認証情報の共有（Cookie共有やセッション管理）の実装コストがかかる。
    *   UIコンポーネントの二重管理やライブラリ共有の手間が発生する。

**結論**: 特段の理由がなければ **Option A (既存統合)** がおすすめです。

## 提案2: 動画ホスティング・配信プラットフォーム

セキュアに（会員限定で）動画を配信するためのバックエンド選定です。

### 1. Cloudflare Stream 【推奨】
開発者フレンドリーでコストパフォーマンスが高い動画配信サービスです。

*   **仕組み**: APIで署名付きURL (Signed URL) を発行し、特定のユーザーのみに再生を許可できます。
*   **メリット**:
    *   Stripe等の会員権限と連動させたアクセス制御が容易（Signed Token利用）。
    *   帯域幅課金ではなく、ストレージ＋再生時間課金で予測しやすい。
    *   Next.js との親和性が高い（Reactコンポーネントあり）。
*   **コスト**: 安価（1000分/月保存で$5 + 再生1000分あたり$1）。

### 2. Vimeo (Pro/Business)
GUIで管理できる既存の動画プラットフォームを利用する方法です。

*   **仕組み**: 動画を「ドメイン制限」設定でアップロードし、自社サイトに埋め込む。
*   **メリット**:
    *   動画の管理画面が使いやすい（非エンジニアでもアップロード容易）。
    *   実装が非常に簡単（埋め込みコードを貼るだけ）。
*   **デメリット**:
    *   ドメイン制限だけなので、HTMLソースを見られるとiframeのURLは特定可能（ただし他サイトでは再生不可）。
    *   APIで細かく制御するには上位プランが必要になる場合がある。

### 3. AWS CloudFront + S3 + MediaConvert
自前でHLS配信環境を構築する方法です。

*   **仕組み**: S3に動画を置き、MediaConvertで変換、CloudFrontのSigned Cookies/URLsで配信。
*   **メリット**:
    *   完全なコントロールが可能。
    *   大規模になれば最もコスト効率が良い可能性がある。
*   **デメリット**:
    *   **実装・構築コストが非常に高い**。
    *   トランスコード設定や署名処理など、メンテナンス負荷が高い。

**結論**: 開発リソースとセキュリティ、UXのバランスから **Cloudflare Stream** が最もモダンで適しています。運用を楽にしたい（コードを書きたくない）場合は **Vimeo** も有力です。

---

## 推奨構成まとめ
*   **アプリ**: `dataviz-auth` (Next.js) 内に `/videos` ルートを追加。
*   **DB/認証**: Supabase (既存) + Stripe (既存)。Userテーブルに「SubscriptionStatus」を持たせる。
*   **動画配信**: Cloudflare Stream。
    *   ユーザーが動画ページにアクセス → Next.js (Server Component) で Stripe/Supabase のサブスク状態確認 → 視聴権限があれば Cloudflare Stream の署名付きトークンを発行 → クライアントプレイヤーに渡す。

## 補足: 動画の不正ダウンロード対策について

ユーザーからの要望により、各プラットフォームにおける「不正ダウンロード（ぶっこ抜き）防止」の観点での比較を追記しました。

| プラットフォーム | 対策レベル | 具体的な仕組み | 評価 |
| :--- | :--- | :--- | :--- |
| **1. Cloudflare Stream** | **高 (推奨)** | **Signed URLs (署名付きURL)** + **HLS/DASH配信**。<br>MP4の直接DLリンクは提供されず、有効期限付きトークンがないと再生不可。一般ユーザーレベルでの保存は非常に困難。 | コストとセキュリティのバランスが最高。<br>「完全なDRM」ではないが、講義動画等の商用利用には十分な強度。 |
| **2. Vimeo (Pro/Business)** | 中 | **ドメイン制限** + プライベートリンク。<br>特定のドメイン(自社サイト)以外での再生を拒否。 | ブラウザ拡張機能やダウンローダーで比較的容易に保存できてしまう。手軽だが防御力は低め。 |
| **3. AWS (S3+CF+MediaConvert)** | **最高** | **HLS暗号化 (AES-128)** または **本格的DRM** (Widevine/FairPlay等) の実装が可能。<br>画面録画すらブロックできる可能性がある(DRMの場合)。 | 実装難易度とコストが非常に高い。ハリウッド映画レベルの保護が必要ならこれ一択だが、通常の会員サイトにはオーバースペック。 |

### 推奨：Cloudflare Stream (Signed URLs)
技術的な実装コストを抑えつつ、実用的なセキュリティレベル（URLの使い回し禁止、直接DL防止）を担保できる **Cloudflare Stream** が「実装しない」段階の計画としても最も理にかなっています。
